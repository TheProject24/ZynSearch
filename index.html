Design intent

What it's optimizing for

ZynSearch isn't trying to be a generic search demo. Search systems tend to become opaque fast — ZynSearch goes the other way, keeping the internal logic legible enough to audit, extend, and trust.

01

Fast enough to feel responsive

Hash-driven postings lookups and bounded top-K collection avoid doing more work than the query needs.

02

Clear enough to reason about

Every transformation — tokens, postings, scores — stays inspectable rather than buried in opaque handlers.

03

Secure enough to serve SDKs

Basic Auth, optional TLS, and structured JSON errors at the server boundary, not bolted on later.

04

Modular enough to embed or deploy

The same core crate runs headless inside a Rust binary or behind a full HTTP/gRPC server.

05

Explicit about its own math

BM25's IDF and TF terms are visible, named, and tunable — never an unexplained black box.

The big picture

One engine, four doors in

The REST API, gRPC service, CLI, and SDKs are different windows onto the same search brain. Every document and every query passes through this same shape.

Core engine philosophy

Four rules, then a file for each job

The core lives in crates/zynsearch-core and is built around keeping ingestion deterministic, search execution explainable, ranking math explicit, and transport concerns entirely out of the engine. That discipline is what lets the project behave like a product instead of a pile of handlers. Tap a card to open it.

Retrieval logic

Two ways to ask "does it match?"

The retrieval pipeline runs in two modes from the same index: a cheap boolean intersection, and a more expressive scored ranking.

  • Analyze the query into tokens, the same way documents were tokenized.
  • Look up each token in the inverted index.
  • Build a candidate document-ID set per token.
  • Intersect the sets — if any term is missing, the result is empty immediately.
  • Map surviving document IDs back to their source paths.
matches(query) = documents(term₁) ∩ documents(term₂) ∩ … ∩ documents(termₙ)
  • Analyze the query into tokens.
  • Collect candidate postings for each token.
  • Compute IDF per term across the corpus.
  • Compute BM25 contributions per posting.
  • Sum scores per document, then sort the top candidates.
score(d,q) = Σ score(t,d) for t in q — "how strongly does it match," not just "does it match."
Sharding & coordination

Split the work, merge the answers

Each document is deterministically assigned to one shard — in the current implementation, by document ID modulo shard count — so the same routing rule can revisit it during search. query_pipeline.rs and searcher.rs coordinate the rest:

→

Parallelizable

Each shard searches independently, so work scales with partitions, not with corpus size alone.

→

No single bottleneck

The coordinator merges results rather than doing all the matching itself.

→

Horizontally scalable later

The deterministic routing rule makes it straightforward to add capacity without re-architecting.

Index shape

Why the inverted index is the right shape

Search engines don't scan every document — that's too slow. They flip the relationship: instead of asking what terms live in a document, they ask what documents contain a term.

Forward index — wrong question

Answers: "what terms are in this document?"

doc_42 → [rust, async, index, shard, http] doc_43 → [token, stem, parser, http] doc_44 → [grpc, proto, stream, http]

Inverted index — right question

Answers: "what documents contain this term?"

http → [doc_42, doc_43, doc_44] shard → [doc_42] proto → [doc_44] token → [doc_43]
Complexity intuition

Doing the least work necessary

Tokenization

Linear in input size.

Postings lookup

Hash-map driven — effectively constant time per term.

Boolean intersection

Depends on the size of the posting lists involved.

BM25 ranking

Proportional to the number of candidate postings actually inspected.

Top-K collection

A bounded heap avoids a full sort of every match.

Server & APIs

Same engine, different doors

The REST layer and the gRPC contract expose the same product idea through different doors — REST for SDKs and simple integrations, gRPC for typed, higher-throughput machine-to-machine traffic.

crates/zynsearch-server/src/http.rs

REST gateway

Translates HTTP requests into engine actions and engine actions back into JSON.

  • Basic Auth
  • Optional TLS
  • Structured error responses
  • SDK-friendly request/response shapes
proto/zynsearch/v1/zynsearch.proto

gRPC contract

The source of truth for typed clients across languages — generated with standard protoc tooling or Buf.

  • Strong typing
  • Cross-language compatibility
  • Clean SDK-generation surface
crates/zynsearch-server/src/main.rs

Transport selection

Boots whichever transport the operator chose — HTTP, gRPC, TCP, or both — and hydrates the engine on startup.

  • --protocol http
  • --protocol grpc
  • --protocol both

SDK surfaces

The same three verbs, in three languages

Each SDK exists to make the same backend feel native — and stays deliberately thin, because the intelligence belongs in the core and the transport layer, not in client wrappers.

sdks/zynsearch-js

JavaScript & TypeScript

Async-first, camelCase methods, JSON-native request and response handling.

  • index() · search() · delete()
  • Best fit: Node.js services, serverless, TS app layers
sdks/zynsearch-py

Python

Synchronous and asynchronous client entry points with minimal ceremony.

  • index() · search() · delete()
  • Best fit: automation jobs, data pipelines, notebooks
sdks/zynsearch-go

Go

A small, direct API that feels like a familiar HTTP wrapper, not a framework.

  • Index · Search · Delete
  • Best fit: microservices, background workers, API adapters
Security & delivery

The recommended production stack

  • HTTPS with certificates.
  • Basic Auth for request identity.
  • REST or gRPC, chosen by integration style.
  • SDKs that talk to the server, never to the core directly.

Why the separation matters

The engine can be reused without networking. The server can evolve without touching scoring math. SDKs can stay idiomatic without duplicating engine logic. The repository stays explainable as it grows — that separation is a major part of the project's identity.
